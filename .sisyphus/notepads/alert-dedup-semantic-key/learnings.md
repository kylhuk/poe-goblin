# Learnings
- Strategy SQL contracts will soon need `semantic_key` in both candidate and backtest surfaces, so tests now codify that alias requirement before runtime catches up.
- Seven packs currently diverge between candidate/backtest identity expressions: `advanced_rare_finish`, `cluster_basic`, `corruption_ev`, `dump_tab_reprice`, `flask_basic`, `high_dim_jewels`, and `rog_basic`.
- Stable-key design guidance from external references aligns with this plan: business identity should be explicit and stable, while timestamps remain ordering metadata only.
- For the seven drifted item-based packs, using `concat(category, ':', base_type, ':', coalesce(price_currency, 'none'))` as both `item_or_market_key` and `semantic_key` keeps scanner and backtest identity parity while excluding volatile refresh fields.
- Latest bulk/market packs now duplicate their stable `item_or_market_key` expressions as `semantic_key`, ensuring every backtest and candidate emit the mandatory column without touching volatile metadata.
- `candidate_from_source_row` now treats `semantic_key` as mandatory canonical identity, while preserving previous `item_or_market_key` and `legacy_hashed_item_or_market_key` values only through `legacy_item_or_market_keys`, `source_item_or_market_key`, and `source_row_json` evidence.
- `build_evidence_snapshot` must overwrite stale source `item_or_market_key` and `legacy_item_or_market_keys` fields with the canonical candidate values; otherwise downstream scanner evidence can still advertise the pre-semantic identity.
- Scanner persistence stays stable across refresh-only row changes when recommendation and alert payloads both route `item_or_market_key` through the candidate semantic key; `source_row_json` and `legacy_hashed_item_or_market_key` can change between scans without changing `alert_id`.
- Backtest detail rows now double-check that `item_or_market_key` stores the canonical semantic identity while `detail_json` keeps the legacy source information, making parity of detail reporting explicit.
- Cooldown regression coverage now includes legacy hashes to show historical entries still block a new semantic candidate, ensuring the compatibility path stays exercised.
- Compatibility hashes are now built from the canonical semantic identity plus its original source key, so scanner and backtest now derive the same stable legacy hash even when metadata-only fields change.
