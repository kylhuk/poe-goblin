# Learnings
- _ops_scanner_recommendations now captures and forwards the optional `cursor` term so the payload contract is locked in before cursor semantics land.
- The scanner recommendation route tests now prove both the full argument set and that payload-raised cursor validation errors turn into `400 invalid_input` responses.
- `scanner_recommendations_payload()` now builds one canonical sort-spec map and reuses it for sort validation, SQL `ORDER BY`, cursor tuple encoding, and seek predicate generation.
- Keyset paging now uses SQL-side filters plus `LIMIT page_limit + 1` to derive `meta.hasMore` and an opaque signature-bound `meta.nextCursor` without Python post-filtering/sorting.
- Scanner recommendations now compute `expected_hold_minutes` and `expected_profit_per_minute_chaos` inside the backend SQL subquery, so payload mapping and `sort=expected_profit_per_minute_chaos` share one canonical hold-duration calculation and keep null per-minute rows last.
- Frontend `api.getScannerRecommendations` now accepts explicit sort/cursor/filter params and surfaces `{ recommendations, meta }` so callers can read `meta.hasMore`/`meta.nextCursor` plus the new `expectedProfitPerMinuteChaos`/`expectedHoldMinutes` fields before wiring granular pagination into the UI.
- Vitest now runs without hitting `ENOSPC` by forcing Node's `tmpdir()` to point at `frontend/.vitest/tmp` via the new `frontend/vitest-ensure-tmp.cjs` hook (loaded through `frontend/.npmrc`), so the shared test suite can execute even though `/tmp` is full inside this container.
- The dashboard path now explicitly requests `sort=expected_profit_per_minute_chaos` with `limit=3` in both backend and frontend call sites, so the card copy about per-minute value matches the actual scanner query.
- The exact `npm --prefix frontend exec vitest run ...` flow executes from the repo root in this worktree, so focused component tests stay portable by using a file-local jsdom pragma plus relative imports/mocks instead of depending on frontend-only alias config.
- Live QA already seeds two scanner recommendations (`cluster_basic` + `bulk_essence`), but `poe_trade.qa_contract.seed()` was only reporting the first non-journal strategy in `qa-seed.json`; fixing the summary/fixture accounting plus forcing `VITE_SCANNER_RECOMMENDATIONS_PAGE_SIZE=1` only in `frontend`'s `qa:dev` flow makes Playwright visibly exercise `scanner-load-more` without changing production pagination behavior.
- A repo-root `vitest.config.ts` now proxies to `frontend/vitest.config.ts` and forces `test.root` inside the frontend folder so `npm --prefix frontend exec vitest run` keeps companion/Bun suites and Playwright specs out of the run.
